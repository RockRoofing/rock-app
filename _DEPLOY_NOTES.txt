DEPLOY NOTES - ROOT CAUSE FOUND: date parser broke both cards (Payless + Avg Days)
==================================================================================

THE SMOKING GUN
---------------
Your Payless diagnostic said: "8 total ... By month: Jan 70: 8." All 8 credit notes
landed in JANUARY 1970. 1970 is the Unix epoch - the tell-tale sign of a broken date
parse.

Root cause: parseXeroDate() in commercial-metrics.js did:
    String(d).match(/\d+/)   // first run of digits
For a normal ISO date "2026-03-15" that grabs "2026" and builds new Date(2026),
which is 2026 MILLISECONDS after 1 Jan 1970 = Jan 1970. It only worked for Xero's
/Date(1710460800000+0000)/ format, not for the plain YYYY-MM-DD dates actually
stored on the invoice/credit-note lines.

This one bug broke BOTH cards:
  - Payless: every credit note bucketed into "Jan 1970", so no real month had any ->
    0 every month, 8 total.
  - Average Days Beyond Terms: due/paid dates parsed to ~1970 too, so the day
    differences were nonsense and got filtered out -> headline blank, no "Avg =".

THE FIX
-------
parseXeroDate() now ONLY extracts the timestamp when the string is genuinely the
/Date(...)/ format; otherwise it parses the string normally (ISO etc.). Verified:
  "2026-03-15" -> 2026-03 (correct)
  "/Date(1710460800000+0000)/" -> 2024-03 (still works)
  paid 2026-03-06 minus due 2026-03-01 -> 5 days (correct)

Also added:
  - API marker bumped to "metrics-v4" (diagnostic line shows "[API: metrics-v4]").
  - Payless diagnostic now reports credit notes with NO valid date, e.g.
    "3 have no valid date (excluded)", so if any genuinely lack a date we see it.
