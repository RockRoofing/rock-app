WHAT THIS DOES
--------------
Ties the chase emails AND Project Details to the Internal Handover Minutes (IHM)
as the source of truth, matched by JOB NUMBER (Xero jobNo <-> IHM projectNo).

Resolution order for every person field:
   commercial override (Project Details)  →  IHM value  →  blank (manual entry).
Commercial overrides are remembered and NEVER written back to the IHM.

NEW: lib/projectPeople.js
  Resolves a project's team (CM/OM/QS/Estimator) + customer contacts + customer QS
  from ops:projects (the IHM store) and portal users (name -> email/phone), with
  the settings.peopleOverride layer on top.

CHASE EMAILS (outstanding-invoices.js)
  • "Hi [name]" now = the CUSTOMER'S QS (the IHM site contact whose role is
    Quantity Surveyor). "To" defaults to that QS's email.
  • Signature now = the LOGGED-IN SENDER:
        <Name>
        <Email>
        <Phone>
        Rock Roofing
    (Templates still sign off with [Rock Roofing QS Name]; that final line is
    expanded into the full signature block on compose.)
  • Reply-to = the logged-in sender.
  • CC quick-add chips now list the resolved Rock CM and each IHM customer contact.

PROJECT DETAILS (project/[id].js + api/project/[id].js)
  • "Team" section: the old free-text "manage team members" area is replaced by
    four role dropdowns — CM, OM, QS, Estimator — each listing ALL PORTAL USERS.
    Defaults come from the IHM (badge: "from IHM"); changing one sets an override
    (badge: "override") that sticks and does not touch the IHM.
  • "Customer Contacts": now a repeatable list (Role/Name/Email/Phone) pulled from
    the IHM site contacts; editing overrides for this project only. The contact
    with role "Quantity Surveyor" is the customer QS used by chase emails.

SEQUENCE (your question): if a Xero project exists BEFORE the IHM is completed,
the fields are blank and you enter them manually. Once the IHM is later completed
with the matching job number, the values auto-populate on the next load (nothing
is copied — resolved live), and any override you set is preserved and wins.

portal-auth.js: the "me" endpoint now returns the sender's phone (enriched from
the stored user record) so the email signature has it.

NOTE ON REFRESH: the chase-email side reads the resolved people from the dashboard
cache (4h). An override set in Project Details shows in chase emails after the
next dashboard sync/refresh; Project Details itself updates immediately.
