CONCERN INVITE FIX — "no meeting attendees" bug

  pages/api/concern-invite.js
   ROOT CAUSE: the invite API was fetching /api/team, /api/tasks, /api/risks over HTTP.
   On the main portal domain the login middleware requires auth for those endpoints, so the
   server-to-server calls (which carry no login cookie) were bounced to the login page and
   returned no data — hence "no attendee emails".
   FIX: read portal users, live tasks and risks DIRECTLY from the database (getPortalUsers,
   getLiveTasks, get('ops:risks')). No internal HTTP calls. Attendees now resolve correctly.

STILL REQUIRED (unchanged):
  - Your Portal User record must have an email address (the invite goes to attendee emails).
  - RESEND_API_KEY set, and FORMS_FROM_EMAIL set to your send.rockroofing.co.uk sender.
