RAMS SIGNING — PHASE 1: Filing restructure + notification badges
================================================================
Deploy target: RockRoofing/rock-app  branch main  (GitHub web editor → Vercel)
Build: `npm run build` → ✓ Compiled successfully (Xero warnings unrelated/harmless)

WHAT THIS DOES
--------------
1. Site App home (siteapp.rockroofing.co.uk) — "View Project Details" is split
   into TWO separate top-level cards:
     • Drawings  (drawings only)
     • RAMS      (immediately after Drawings — RAMS only)
   Each opens straight to its own project picker → file list (no tab switcher).

2. RAMS home card shows a RED notification badge = the number of RAMS documents
   the logged-in operative has NOT signed, across the projects they're assigned
   to (respects projectAccess: 'all' or a list). Same badge style as
   Issues Log / Live Tasks.

3. Inside RAMS, each document is labelled "● Not signed" (red) or "✓ Signed"
   (green) for the current operative. (The actual "Sign RAMS now" flow arrives
   in Phase 2 — this phase just surfaces the status + counts.)

4. Deliveries home card shows a RED badge = number of scheduled-but-not-delivered
   deliveries on the operative's assigned projects. Inside Deliveries, each
   project in the picker shows its own red badge with its scheduled count.

DATA MODEL INTRODUCED (used by Phases 2–5)
------------------------------------------
• lib/db.js: new helpers getRamsSignatures(projectNo) / saveRamsSignatures(...)
  Store: ops:rams-signatures:<projectNo> = {
    [fileId]: { [opId]: { name, date, signedAt, statement } }
  }
  Keyed by the RAMS file's id. Because /api/project-files mints a NEW id on
  every upload, a re-uploaded RAMS (new version) has no signatures and is
  "Not signed" again automatically — this is how version-reset will work.

NOTES / EXPECTED BEHAVIOUR
--------------------------
• Until anyone has signed anything, every RAMS shows "Not signed" and the RAMS
  badge = total current RAMS docs across your assigned projects. That's correct
  for Phase 1 — signing lands in Phase 2.
• Badge fetch uses user.id from the Site App session (sessionStorage
  'ops_operative'). If an operative record has no id, the RAMS-signed lookup
  falls back to "not signed" (safe default) but the count still works.
• No cron changes. No portal RAMS Matrix changes yet (Phase 3 adds PS state +
  stage pipeline).
