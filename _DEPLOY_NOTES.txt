DELIVERIES — the real cause of broken attachment images

  WHAT WAS DIFFERENT ABOUT DELIVERIES:
  The Site App Deliveries page uploaded proof photos using FormData:
        const fd = new FormData(); fd.append('file', file)
        fetch('/api/upload-file', { method:'POST', body: fd })
  But /api/upload-file expects RAW file bytes with the name/type in x-filename / x-content-type headers
  (bodyParser is off). So the delivery photos were stored as the multipart-form wrapper with content-type
  multipart/form-data — i.e. a corrupted, non-viewable "image". That's why View showed a broken icon and
  Download saved a file that isn't a real JPEG. Procurement / Risk Log / Live Tasks upload via
  RowAttachments (raw bytes + headers), which is why THEY always worked.

  pages/forms/deliveries.js
   - handleFiles now uploads RAW bytes with the correct headers (matching RowAttachments), so delivery
     proof photos are stored as proper images and render in the viewer everywhere.

  IMPORTANT: this fixes NEW uploads. Any photo uploaded BEFORE this fix (e.g. the current 45361.jpg) was
  stored corrupted and will still show broken — please re-upload those few after deploying.

new correctly-typed images and any legitimately octet-stream ones.)
