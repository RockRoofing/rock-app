Wages sync — stop duplicates accumulating
==========================================
Deploy target: RockRoofing/rock-app  branch main
Build: ✓ Compiled successfully
Single file: pages/api/sync-wages.js

DEPLOY THIS BEFORE RE-SYNCING WAGES.

THE BUG
-------
On each wage sync, the old code kept every existing line that was before the window
OR had no date:
    const older = existing.filter(l => !l.date || l.date < winStr)
Undated lines can never be "in the window", so they were re-kept on EVERY sync while
fresh dated wages were added alongside them — duplicating labour (e.g. the repeated
£20,172.73 "Labour cost upto 30/04/2026 to book under tracking categories" and
£920.52 "June 26 Wages" rows on Russell Hill).

THE FIX
-------
The wage bucket is now rebuilt cleanly on every sync:
 • Keep genuine OLDER dated history (before the window) — nothing is lost.
 • Add the freshly-fetched in-window lines.
 • DE-DUPLICATE by Xero JournalLineID (fresh wins); fall back to a content key
   (date+amount+description+reference) when no id is present.
 • DROP legacy lines that have no date AND no Xero line id — these were the orphaned
   adjustments that caused the pile-up. Real Xero lines always carry a JournalLineID,
   so nothing legitimate is removed.
Applied to both the per-project buckets and the untagged (unassigned) bucket.

AFTER DEPLOY
------------
1. Deploy this file.
2. Re-run the Wages sync (Bookkeeping page or the commercial SyncBar) to repopulate
   the wages you cleared — it will now come in without duplicates.
3. Re-check Russell Hill: the Budget Tracker all-time labour should equal the sum of
   the actual dated wage lines (no doubling).

The diagnostic endpoint (pages/api/cost-reconcile.js) can be deleted once you're happy
— it's read-only and unused by the app.
