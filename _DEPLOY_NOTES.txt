Bookkeeping — FIX: sync dropped 9 of 16 June invoices (Xero rate-limit)
======================================================================
Deploy target: RockRoofing/rock-app  branch main
Build: npm run build -> ✓ Compiled successfully
Deploy, then re-run Sync Invoices.

THE BUG
-------
Your Xero Sales export has 16 June invoices (£258,124.71) but the app showed only
7 (£43,775.86). Missing incl. the big ones (Winnersh £93k, Christchurch £30k,
Market Drayton £34k).

Cause: the sync re-fetched EACH invoice individually to read tracking, ~80ms
apart. That's ~750 calls/min — Xero's limit is 60/min. After ~60 calls Xero
returned 429 (rate limited); the code treated a failed fetch as "skip" and
SILENTLY DROPPED those invoices. So over half of June vanished.

THE FIX
-------
• Read line-item TRACKING straight from the Invoices LIST response (one paged
  call returns 100 invoices WITH their tracking) — no per-invoice re-fetch for
  the normal case. This removes the flood of calls that caused the rate-limiting.
• Only if a specific invoice row comes back with no line items does it re-fetch
  that ONE invoice — and even then it honours Xero's 429 "Retry-After" with
  backoff and retries, so an invoice is NEVER silently dropped.
• List paging spaced to stay well under 60/min; 429 on a list page waits & retries
  the same page.

RESULT
------
After deploying, re-run Sync Invoices. June should now return all 16 invoices
(~£258k), matching your Sales export. Header "X matched, Y unassigned (of Z)" — Z
should jump to the full count.
