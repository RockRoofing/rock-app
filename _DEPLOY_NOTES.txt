WHAT CHANGED vs the first Phase 3
--------------------------------
1. The DIRECTOR NOW SIGNS IN THE SITE APP (not the Portal). The Portal
   "RAMS Approvals" page and its H&S nav link are removed.
2. New ADMIN AREA: "RAMS Director" tab (Admin nav, placed BEFORE "App
   Improvements"). Pick the Director from Portal Users whose job role is
   "Director". Stored as ops:rams-director = { name, email }.
3. The Site App recognises the Director by EMAIL: when the logged-in Site App
   user's email == the designated Director's email, RAMS at the Director stage
   show an "Approve as Director" button (read-to-bottom -> approve & sign, same
   as the CM). The Director auto-signs onto the RAMS with the operative statement.
   IMPORTANT: the chosen Director must also be a Site App user with the SAME email.

THE CHAIN (unchanged, strict, sequential)
-----------------------------------------
  CM (Site App) -> Director (Site App, designated in Admin) -> Site Manager
  (tokenised email, no login) -> Operatives.
  No stage unlocks until the previous is signed. Operative "Sign RAMS now" stays
  locked (with a "waiting on <stage>" note) until the chain reaches operatives.

ADMIN -> RAMS DIRECTOR
----------------------
- New page /admin/rams-director (admin-gated via AdminShell).
- Lists Portal Users with jobRole === 'Director' (that job role already exists
  in Portal Users). Choose one and Save. "None" clears it.
- If no Director is set, the chain will stall at the Director stage (the page
  warns about this).

RAMS MATRIX + GANTT (unchanged from first Phase 3)
--------------------------------------------------
- Matrix: derived Yes (green) / PS (orange) + key + CM › Director › Site Manager
  › Operatives pipeline under each project name (current red, done green).
- Gantt "Installer(s) not on RAMS": only "Yes" counts; "PS" still warns.

DATA MODEL
----------
- ops:rams-director = { name, email }                      (NEW — Admin setting)
- ops:rams-approvals:<projectNo>[fileId] = { stage, cm, director, siteManager,
    siteManagerEmail, siteManagerName, token, ... }
- ops:rams-token:<token> = { projectNo, fileId }

SECURITY
--------
- director-approve verifies the caller's email == the designated RAMS Director
  (from ops:rams-director). No portal session needed (it's a Site App action).
- set-RAMS-Director (POST /api/rams-director) is admin-only (portal session).
- Site Manager link: unguessable 24-byte token; re-issued if recipient changes.

TEST (end to end)
-----------------
1. Upload a RAMS to a project (chain starts at CM).
2. Site App as the CM -> RAMS -> "Approve as Contracts Manager".
3. Site App as the Director (email matches Admin setting) -> RAMS ->
   "Approve as Director".
4. Site App as the CM -> "Send to Site Manager" (pick/confirm email) -> send.
5. Open emailed link -> approve as Site Manager. Operatives unlock.
6. Site App as an operative -> "Sign RAMS now".
7. Portal RAMS Matrix: PS -> Yes; pipeline goes green when all signed.
8. Gantt: unsigned installer shows "not on RAMS" for PS; clears at Yes.
