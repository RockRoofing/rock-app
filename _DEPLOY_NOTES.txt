Valuation date — use manual per-month dates, not end-of-month
=============================================================
Deploy target: RockRoofing/rock-app  branch main
Build: ✓ Compiled successfully
Single file: lib/db.js

THE PROBLEM
-----------
Projects with manually-entered per-month valuation dates (varying each month, no
fixed "valuation day") showed a margin mismatch: the project page's dropdown listed
the correct dates, but the DEFAULT/effective valuation date used for the figures
ignored them and fell back to end-of-month / all-time. So Project Details and the
EOM report didn't line up.

THE FIX
-------
getEffectiveValuationDate() now also reads settings.dateOverrides. When there's no
fixed valuation day, it uses the MOST RECENT PAST override date (the date actually
entered in the table). Future-dated overrides are ignored. This feeds:
  • the project/[id] page figures + WIP (costsAfterDate)
  • the WIP sync cron
  • the project report
so they now match the valuation dates in the table and reconcile with the EOM report
(when set to the same month/date).

Validated: overrides {28 May, 24 Jun, 31 Jan 2027} -> picks 24 Jun (most recent past;
future excluded).

AFTER DEPLOY
------------
Open the project you were looking at — its default valuation date should now be the
latest date from its table (not end-of-month), and the Project Details margin should
match the EOM report for that same date.
