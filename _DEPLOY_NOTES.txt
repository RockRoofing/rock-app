WHAT THE DIAGNOSTIC REVEALED
----------------------------
1. The date IS in the list, but in Microsoft-JSON format: "/Date(1551312000000+0000)/".
   Our old code checked for YYYY-MM-DD, which never matched -> date came out null.
   => THAT is why wages had no dates.
2. The list ALREADY includes JournalLines (hasLines:true, lineCount:6). So the
   per-item fetch-by-ID was pointless -> THAT is why it was slow.
3. The list returns OLDEST-first (Feb 2019 on page 1) -> paging through all history
   made it slower still.

THE FIX
-------
• parseXeroDate() converts "/Date(ms)/" -> "YYYY-MM-DD"  (dates now populate)
• Read JournalLines straight from the list — NO per-item fetch  (fast)
• Request order=Date DESC (newest first) and STOP once a whole page is older than
  the window  (for a 6-month sync it stops after 1-2 pages)
• Filter to the window in code using the parsed date

RESULT
------
Sync should now be quick AND every wage line has its correct date, filterable by
month. June should show the individual project-tagged 320 lines with dates.
