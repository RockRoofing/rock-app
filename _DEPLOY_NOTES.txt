THE BUG
-------
The Bills importer's merge step referenced LABOUR_ACCOUNT_CODES, which isn't
defined in that file (it lives in the wages importer / deep-sync). So the Bills
upload threw "LABOUR_ACCOUNT_CODES is not defined" and stopped BEFORE saving —
meaning that upload's bills were NOT stored. (Previously-stored bills were
unaffected; only the failing upload didn't land.)

THE FIX
-------
Recompute labour vs materials from each line's existing `type` field
('Labour'/'Materials'), which is set at parse time using your Account
Categorisation config. This both fixes the crash AND makes the split correctly
respect any Labour/Materials/Ignore settings you've configured (rather than a
hardcoded code list).

Only the Bills importer had this bug. Wages importer (all-labour) and deep-sync
(which defines the constant) were fine.

ACTION AFTER DEPLOY
-------------------
Re-upload the Bills CSV — the upload that showed the error didn't save, so those
bills need re-importing. It will merge/dedupe as normal (no duplicates).
