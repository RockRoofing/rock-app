NOTIFICATIONS (Resend — uses RESEND_API_KEY + FORMS_FROM_EMAIL)
---------------------------------------------------------------
All best-effort (never block the underlying action). New helper: lib/ramsNotify.js.

1. ALLOCATED TO A PROJECT → the operative gets an email the FIRST time they're
   added to a project on the Gantt (not on every day-cell). Dedupe store:
   ops:rams-notified:allocation. If they're later fully removed from the project,
   the dedupe clears so a future re-allocation notifies again.

2. NEW PROJECT ADDED → Site App users with "All projects" access are emailed that
   they now have access. Fires when an ACTIVE project is created (manual-add or
   IHM). Dedupe: ops:rams-notified:new-project (won't re-send on edits/re-saves).

3. RAMS UPLOADED → operatives who can access that project are emailed that RAMS
   are ready to sign. Fires from /api/project-files when a file with
   category 'rams' is added (this is how the portal uploads RAMS).

4. REMINDERS EVERY 2 DAYS → for RAMS whose chain has reached the operatives
   stage, each accessing operative who hasn't signed is reminded, at most once
   per 2 days per document. Self-throttled via ops:rams-notified:reminder.

CRON (no new slot)
------------------
- New endpoint /api/cron/rams-reminders (exports runRamsReminders). Can be hit
  with ?force=1 to test (ignores the 2-day throttle).
- It is invoked DAILY from the existing /api/cron/hs-expiry-email cron (07:00),
  alongside the forms + deliveries jobs. Because runRamsReminders self-throttles
  to every 2 days per recipient+document, running it daily yields 2-day reminders.
