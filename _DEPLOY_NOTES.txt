RAMS — FIXES v2: Site Manager link, CM email, dup email, badges, matrix pipeline
================================================================================
Deploy target: RockRoofing/rock-app  branch main
Build: npm run build -> ✓ Compiled successfully
Deploy on top of the previous RAMS fixes.

1) SITE MANAGER CAN NOW APPROVE (was: stuck — link needed a Site App login)
--------------------------------------------------------------------------
The Site Manager approval page (/rams-approve?token=...) is a NO-LOGIN page on
the MAIN PORTAL domain (app.rockroofing.co.uk) — it always was, but the email
link was being built from the request host, which was siteapp.rockroofing.co.uk
when the CM sent it from the Site App (that needs a login -> dead end).
FIX: the email link is now ALWAYS built against the portal domain
(app.rockroofing.co.uk), overridable with env RAMS_APPROVE_ORIGIN.
So the customer's Site Manager:
  - gets the email (RAMS attached) → clicks "Review & approve RAMS"
  - lands on the public page (no login), reviews the PDF, enters their name
  - clicks Approve → operatives unlock.
No customer login is ever required. If you have a preferred public URL, set
RAMS_APPROVE_ORIGIN in Vercel (defaults to https://app.rockroofing.co.uk).

2) DUPLICATE CUSTOMER EMAIL (was: sent twice)
---------------------------------------------
set-site-manager now guards against a duplicate send: it won't email the same
recipient for the same RAMS again within 30 seconds (covers double taps /
retries). "Resend / change" after that window still works normally.

3) CM EMAIL ON RAMS UPLOAD (was: not sending)
---------------------------------------------
Cause: the CM was only matched to a SITE APP user by name; if the CM wasn't a
Site App user (or the name didn't match), no email was found.
FIX: findProjectCM now falls back to the matching PORTAL user's email, so the
"RAMS to approve" email reaches the CM even if they aren't a Site App user.
(Best results: keep the project's Contracts Manager name consistent with the
CM's user record.)

4) NOTIFICATIONS / BADGES ON SITE APP HOME + RAMS PROJECTS (was: nothing showing)
--------------------------------------------------------------------------------
The badge is now ROLE-AWARE — it counts RAMS where THIS user actually has an
action to take:
  • CM (their project)  → RAMS waiting at the CM stage
  • Director            → RAMS waiting at the Director stage
  • Operative           → RAMS at the operatives stage they haven't signed
So a CM sees a red badge when a RAMS needs their approval, the Director when it's
their turn, and operatives once it's unlocked for them. (In your screenshot the
RAMS was at the Site Manager stage — at that point no in-app user has an action,
so a zero badge there is correct; it will light up for operatives once the Site
Manager approves.)
The badge fetch now passes accessLevel/email/name so the API can tell who you are.

5) RAMS MATRIX TIMELINE (was: still wrong)
------------------------------------------
The matrix pipeline now uses the EXACT same rendering logic as the Site App
pipeline you confirmed works (pure stage-driven, ✓ on completed stages, current
stage red+bold, upcoming grey). Combined with the earlier fix that excludes the
CM/Director auto-signatures from the operative signer set, the line now matches
the Site App: e.g. ✓ CM  ✓ Director  Site Manager(red)  Operatives(grey).
