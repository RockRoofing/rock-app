THE CHAIN (strict, sequential — no stage until the previous is signed)
----------------------------------------------------------------------
  CM (Site App) -> Director/Carl (Portal) -> Customer Site Manager (email) -> Operatives
- CM and Director APPROVE (safe method / risk-assessed) AND auto-sign onto the
  RAMS with the same statement operatives use (they appear in signatures/matrix).
- Operatives' "Sign RAMS now" (Phase 2) is now LOCKED until the chain reaches
  them; it shows which stage it's waiting on.

WHAT EACH PERSON SEES
---------------------
• CM (Site App → RAMS): each RAMS shows the pipeline line and, at the CM stage,
  an "Approve as Contracts Manager" button (read-to-bottom → sign). After the
  Director approves, the CM gets "Send to Site Manager for approval" — a picker
  that pulls candidates from the project's Site Contacts (any contact whose role
  contains "manager" is pre-selected), and allows changing or manually entering
  a name + email. Sends a tokenised email with the RAMS attached.
• Director/Carl (Portal → Operations → H&S → RAMS Approvals): NEW page listing
  every RAMS awaiting Director approval. Review (read-to-bottom) → approve & sign.
  Identified by email DIRECTOR_EMAIL (default carl@rockroofing.co.uk); admins may
  also approve (fallback). Set DIRECTOR_EMAIL in Vercel env to Carl's exact email.
• Customer Site Manager: receives the email, clicks the unique link to a no-login
  page (/rams-approve?token=...) showing the RAMS + an Approve button. They enter
  their name (pre-filled if known); date auto-fills. On approval, operatives unlock.

RAMS MATRIX (Portal → H&S → RAMS Matrix) — now DERIVED from real data
--------------------------------------------------------------------
• Cells: "Yes" (green) = signed all current RAMS; "PS" (orange) = chain has
  reached operatives and this person hasn't signed; blank = not yet / no RAMS.
• KEY added at the top (Yes – Signed RAMS · PS – Pending Signature).
• Under each project name: the pipeline CM › Director › Site Manager › Operatives.
  Current stage = red+bold; completed = green; all signed = whole line green.
• Cells are now read-only (auto from signatures); the old manual toggle is retired
  (its store is kept but unused). Matching is by operative NAME (roster name vs
  signed name) — keep roster names consistent with Site App user names.

GANTT (Planning) — "Installer(s) not on RAMS" warning
-----------------------------------------------------
Now treats ONLY "Yes" (fully signed all current RAMS) as signed. "PS" (pending)
STILL triggers the warning, per spec.

VERSION RESET (unchanged from Phase 2, applies here too)
--------------------------------------------------------
New RAMS upload = new file id = fresh approval chain (starts at CM) + no
signatures. Matrix/pipeline/badges all reset automatically.

DATA MODEL
----------
• ops:rams-approvals:<projectNo>[fileId] =
    { stage, cm, director, siteManager, siteManagerEmail, siteManagerName,
      token, startedAt, updatedAt }
    stage ∈ 'cm'|'director'|'site-manager'|'operatives'|'complete'
    cm/director records include a signatureImg (they auto-sign).
• ops:rams-token:<token> = { projectNo, fileId }  (Site Manager link lookup)

SECURITY
--------
• Site Manager link uses a 24-byte random token (unguessable, one per recipient;
  re-issued if the CM changes recipient — old link dies).
• director-approve requires a portal session AND email == DIRECTOR_EMAIL (or admin).
• sm-approve requires a valid token. cm-approve is trusted from the Site App
  (consistent with the rest of the Site App, which has no server session).

ENV VARS
--------
• DIRECTOR_EMAIL  (optional) — Carl's portal email. Default carl@rockroofing.co.uk.
• Uses existing RESEND_API_KEY + FORMS_FROM_EMAIL for the Site Manager email.
