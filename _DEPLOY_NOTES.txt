ATTACHMENTS — View shows the actual image (MIME-type fix)

  Diagnosis: the overlay opened but showed a broken-image icon. The manually uploaded delivery photos were
  stored on the blob with content-type "application/octet-stream" (the browser sent no type on upload), and
  browsers refuse to render <img> for a non-image content type — while Download still worked (it just saves
  the bytes). Drawings didn't hit this because those files were stored with correct image types.

  pages/api/download.js
   - When serving inline, if the stored content-type isn't an image/PDF, it now derives the correct MIME
     type from the file extension (jpg/jpeg/png/gif/webp/bmp/heic/heif/pdf) so <img>/<iframe> render it.

  components/RowAttachments.js
   - The viewer loads images/PDFs via /api/download?inline=1 (correct MIME type, same origin). If that ever
     errors, the <img> falls back to the raw blob URL.

Affects Deliveries, Procurement, Risk Log, Live Tasks. Deploy BOTH files.
