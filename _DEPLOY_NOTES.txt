Deploy on top of the previous RAMS fixes.

1) BADGES / MATRIX NOW USE ONLY THE CURRENT RAMS REVISION (root-cause fix)
-------------------------------------------------------------------------
THE recurring matrix bug: the matrix rolled up the stage across ALL RAMS files
for a project and took the LEAST-advanced one. An OLD revision still sitting at
the CM stage dragged the whole line back to "CM" — while the Site App (which
only ever looks at the newest revision) correctly showed "Site Manager".
FIX: the matrix and the badges now consider ONLY the newest RAMS file per
project (sorted by uploadedAt), exactly like the Site App. The pipeline now
matches the Site App everywhere.
This also fixes your revision point: uploading a new RAMS supersedes the old one
for stage, signatures, badges and the matrix — old-version notifications no
longer linger (e.g. the Director won't see a badge for a version he can no
longer access).

2) DIRECTOR "TO APPROVE" EMAIL — more robust
--------------------------------------------
notifyDirectorToApprove now: (a) falls back to a Portal user with jobRole
"Director" if ops:rams-director has no email, and (b) logs a clear error if the
send fails (check RESEND_API_KEY / FORMS_FROM_EMAIL / the director's email).
If it still doesn't arrive, the cause is almost certainly email config or the
director's stored email — the deploy notes flag exactly what to check.

3) SITE MANAGER: "DO NOT APPROVE — REQUIRES EDITS"
--------------------------------------------------
The no-login Site Manager page now has a "Do not approve — requires edits"
option. The Site Manager enters their name + a description of the edits; on
submit the RAMS is NOT approved and an email is sent to BOTH the project's
Contracts Manager AND the designated Director with the edit notes. (Make the
edits, re-upload the RAMS to restart the chain.)

4) DIRECTOR SIGN = SIGN + AUTO-SEND (already in v3, included)
------------------------------------------------------------
When the Director signs, the customer's Site Manager email is sent AUTOMATICALLY
(recipient chosen by the CM at CM approval). No separate "send" step.

5) SITE APP USERS  ⇄  H&S OPERATIVES  — now one list
---------------------------------------------------
• Site App Users (Admin) gains COMPANY and TRADE(S) fields. ALL fields are now
  mandatory when adding a user (name, role, mobile, email, company, ≥1 trade).
• The H&S → Operatives page is now READ-ONLY and is automatically populated from
  Site App Users (it shows name, email, phone, company, trade). Add/Edit/Delete
  there is removed — manage everyone under Admin → Site App Users.
• /api/operatives now returns a projection of Site App Users, so the Planning
  Gantt and RAMS Matrix draw their installer list from the same single source.

  ⚠ IMPORTANT — allocations reset: because operatives now use Site App User IDs,
  any EXISTING Gantt labour allocations tied to the old separate roster will no
  longer resolve. You confirmed this is fine (few/none yet). New allocations use
  the unified users going forward. The old roster store (ops:operatives-roster)
  is simply no longer read.

  ACTION: re-add your installers as Site App Users (with company + trade) if they
  aren't already there; they'll appear in Operatives, the Gantt and the Matrix.
