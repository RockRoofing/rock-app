BEHAVIOUR
  - Issues do NOT appear in the Forms list or the per-project Project Forms tab.
  - Issue photos DO appear in Project Images (viewable like any other form's photos).
  - Form fields (* mandatory): *Project (dropdown), Created Date (auto), Project Address (auto),
    Form Created By (auto from operative), Issue ID (auto ISS-0001), *Issue Name, *Issue Type
    (multi-select: Design, Quality, H&S, Delay/Programme, Deliveries/Materials, Water ingress,
    Damage to our works, Interface issue, Access, Weather, Customer/Main Contractor, Workmanship,
    Other+description), *Issue Description, *Attach Photos.
  - Tracker columns: Send to Customer? | Project | Created | Created By | Issue Name | Issue Type |
    Resolved Date | Comments | Attachments | Edit | Delete. Inline editable. Sort + 50/page.
  - Send to Customer? state machine:
      (nothing) -> "⚠ Needs actioning"
      Yes – Send -> opens pop-out of IHM customer contacts (remove line-by-line / add / Send PDF)
      Yes but not sent -> red "⚠ Not sent yet" + "Send now" + "Sent manually?" (clears the warning)
      No – Requires edits -> hold state
      No – Do not send -> line clear
  - Resolved Date set -> row turns green / marked resolved.
  - Edit opens a click-off-safe window (backdrop won't close it): Save / Download PDF /
    Send to Customer / X.
  - Notification email to CM/Ops/QS contains a button linking to the portal (desktop) Issues
    tracker (?issue=<id> deep-links straight to the issue) or the Site App (mobile).

REQUIRES (already used elsewhere): RESEND_API_KEY + FORMS_FROM_EMAIL env vars.
Recipients resolve from IHM team names -> Portal Users -> emails; customer contacts from IHM siteContacts.

NOT DONE YET (per your instruction): do NOT send issues to the customer automatically — sending is
a manual action in the tracker. Client-send wording can be refined before first real use.
